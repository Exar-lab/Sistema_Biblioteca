# Proposal: Oracle Stored Procedures + Repository Layer

**Change ID**: `oracle-stored-procedures`
**Project**: `sistema_biblioteca`
**Delivery**: chained PRs, stacked-to-main (3 PRs)

---

## 1. Intent

### Problem
The hexagonal architecture has the domain (`app/domain/models/`) and the application ports (`app/application/ports/`) defined, but `app/infrastructure/repositories/` is empty. There is no concrete persistence layer, so application services cannot be wired and the API surface cannot be built on top of real data. Additionally, `database/oracle_schema.sql` only ships tables, triggers, and FK constraints — there are no explicit sequences, no stored procedures, and no `SYS_REFCURSOR` types, which is a hard requirement for the academic deliverable that mandates writes through PL/SQL.

### Why now
Every downstream phase (services, routes, templates, auth) is blocked on having a working persistence layer. Until repositories exist and satisfy the Protocol ports, services cannot be implemented, and without services the FastAPI controllers have nothing to call. The Oracle stored-procedure requirement is a course constraint — implementing the repositories against ORM-only writes first and migrating later would mean throwing the work away.

### Success criteria
- `database/oracle_schema.sql` can be applied to a fresh `BIBLIOTECA` schema AND to the existing one without errors (idempotent additions).
- Every Protocol in `app/application/ports/` has a concrete implementation in `app/infrastructure/repositories/` that:
  - performs **writes** (INSERT / UPDATE / DELETE) through stored procedures
  - performs **reads** (SELECT) through SQLAlchemy ORM
- `python -m pytest` runs green for the new repository tests against a stub/mock layer (no live Oracle required in CI).
- `python -m compileall app main.py` succeeds.

## 2. Scope

### In scope — exactly the agreed 3-PR split

**PR 1 — Oracle schema foundation** (`database/oracle_schema.sql`)
- Add explicit `SEQUENCE` objects for every table currently using `GENERATED BY DEFAULT AS IDENTITY` (kept for backward compatibility; sequences used by procedures).
- Add `SYS_REFCURSOR` return types where needed for any read-back operation inside procedures.
- Add CRUD stored procedures for: `roles`, `library_users`, `categories`, `authors`, `books` (+ `book_authors` link), `loans`, `returns`.
  - Organized as one PL/SQL package per aggregate (`pkg_roles`, `pkg_users`, `pkg_categories`, `pkg_authors`, `pkg_books`, `pkg_loans`, `pkg_returns`).
  - Each package exposes `p_insert`, `p_update`, `p_delete` at minimum; books/loans get the extra operations they need (link author, register return).
- All DDL additions wrapped in idempotency guards (`BEGIN EXECUTE IMMEDIATE '...'; EXCEPTION WHEN OTHERS THEN IF SQLCODE != -955 THEN RAISE; END;`) for sequences; `CREATE OR REPLACE PACKAGE` / `PACKAGE BODY` for procedures.

**PR 2 — Python repositories** (`app/infrastructure/repositories/`)
- One file per aggregate: `book_repository.py`, `author_repository.py`, `category_repository.py`, `loan_repository.py`, `return_repository.py`, `role_repository.py`, `user_repository.py`.
- Each class implements the matching Protocol from `app/application/ports/`.
- **Writes** call stored procedures via `session.execute(text("BEGIN pkg_x.p_insert(:p1, :p2, ...); END;"), {...})` or via cursor `callproc` when bind direction (OUT params) is needed.
- **Reads** use SQLAlchemy 2.0 ORM (`session.execute(select(Model)...)`) directly against the domain models.
- Transaction boundaries owned by the service layer / unit-of-work — repositories accept a `Session` and do not commit.

**PR 3 — Regression tests** (`tests/unit/`, `tests/integration/`)
- Pytest tests for every repository implementation, mocking `Session.execute` to assert the correct PL/SQL block + bind params are issued for writes, and asserting ORM query shape for reads.
- Smoke tests for any application service that gets wired during PR 2 (only if a service is required to satisfy a port contract — otherwise services are deferred).
- Run `python -m compileall app main.py` as part of the CI test step.

### Out of scope
- FastAPI route handlers / controllers.
- Jinja2 templates and any presentation concern.
- Application service orchestration beyond what is strictly needed to demonstrate the repositories work end-to-end.
- Authentication, authorization, session management.
- Live Oracle integration tests in CI (deferred; covered by mocked unit tests for now).
- Data migrations of existing rows (schema additions are pure DDL extensions).

## 3. Approach

### Schema additions (PR 1)
- **Sequences first, identity columns kept**: existing `GENERATED BY DEFAULT AS IDENTITY` columns are preserved so currently-running inserts (if any) keep working. New sequences (`seq_books`, `seq_authors`, etc.) are referenced explicitly inside procedures so the procedure controls the ID — this is what the course rubric expects.
- **Idempotent DDL**: every `CREATE SEQUENCE` wrapped in a PL/SQL block that swallows `ORA-00955` (object already exists). Packages use `CREATE OR REPLACE`, which is idempotent by definition.
- **Read-back via REF CURSOR**: procedures that need to return rows (e.g. `p_get_book_with_authors`) declare an `OUT SYS_REFCURSOR` parameter; the Python side fetches it through `cursor.var(oracledb.CURSOR)`.

### Repository pattern (PR 2)
- **Hexagonal compliance**: each repository class depends on the Protocol from `app/application/ports/` (structural typing — no inheritance required, but we keep the explicit `: BookRepositoryPort` annotation for clarity).
- **Write path**: thin wrapper that builds the PL/SQL anonymous block, binds parameters by name, executes via `session.execute(text(...), {...})`. For OUT params or REF CURSORs we drop to the raw DBAPI cursor via `session.connection().connection.cursor()`.
- **Read path**: standard SQLAlchemy 2.0 `select()` against the existing ORM models in `app/domain/models/`. Eager loading (`selectinload`) for relationships when the port returns aggregates.
- **No session ownership**: constructor takes `Session`; no `commit()` / `rollback()` inside repositories.

### Testing (PR 3)
- **Unit tests** mock the `Session` and assert:
  - the exact PL/SQL text passed to `text(...)`
  - the exact bind dict
  - for reads, the constructed `select(...)` (compared via `.compile()` string)
- **No live DB required** for CI — that keeps the build deterministic.
- **Optional integration tests** under `tests/integration/` are scaffolded but skipped by default with `pytest.mark.skipif(not ORACLE_DSN, ...)` so a developer with a local Oracle can opt in.

## 4. Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| No live Oracle in CI → procedures could be syntactically wrong and pass mocked tests | High | High | PR 1 includes a `sqlplus`-style smoke script doc; reviewer with local Oracle XE runs `@oracle_schema.sql` before merge |
| `oracledb` thin mode doesn't support some PL/SQL features (e.g. `DBMS_OUTPUT`) | Medium | Low | Avoid DBMS_OUTPUT in procedures; use OUT params for any feedback. Thin mode supports REF CURSOR and named binds, which is all we need |
| Idempotency block for sequences swallows unrelated errors | Low | Medium | Narrow exception filter to `SQLCODE = -955` only and `RAISE` otherwise |
| Adding sequences alongside identity columns causes confusion about who owns the ID | Medium | Medium | Document in schema header: "procedures use sequences; identity kept only for ad-hoc inserts during development" |
| Repository tests become tautological (asserting we wrote the SQL we wrote) | Medium | Low | Keep assertions on bind-param shape and on the procedure name being called — the actual SQL string is part of the contract being tested |
| Port Protocols may need adjustment once concrete repos are written | Medium | Low | Allowed within PR 2; flag any signature change in the PR description |

## 5. Rollback Plan

Each PR is independently revertible (stacked-to-main):

- **Revert PR 3**: `git revert <pr3-sha>` — deletes test files only. No runtime impact.
- **Revert PR 2**: `git revert <pr2-sha>` — deletes `app/infrastructure/repositories/*.py`. Services (none exist yet) lose their concrete implementations; ports remain.
- **Revert PR 1**: `git revert <pr1-sha>` — removes the schema additions from the SQL file. Already-applied sequences/packages in a deployed DB must be dropped manually with `DROP SEQUENCE seq_x` / `DROP PACKAGE pkg_x` — document this in the PR 1 description as a manual rollback step.

No data migration → no data rollback needed.
